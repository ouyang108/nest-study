# 4如何理解nest中的.service和.controller的区别
## controller 控制器:前端服务员
唯一职责：处理http请求，并将结果返回给客户
服务员不需要知道牛排是怎么煎出来的、需要腌制几分钟，他只需要做三件事：
 1.迎接客人：通过路由（如 @Get('users')）知道客人要干嘛。
 2.检查菜单和洗手：通过 Pipe 验证客人传过来的参数（DTO）对不对（比如买酒是不是满 18 岁）。
 3.点菜传单：把验证好的数据，转交给后厨的 Service（服务），然后站在那里等服务员把菜做好，最后打包（返回 JSON 响应）给用户。

```javascript
@Controller('books') // 路由前缀：http://localhost:3000/books
export class BooksController {
  // 1. 通过构造函数，把后厨（Service）叫过来
  constructor(private readonly booksService: BooksService) {}

  @Post()
  create(@Body() createBookDto: CreateBookDto) {
    // 2. 收到请求，进行入参校验（class-validator 会在后台默默工作）
    // 3. 不在 Controller 里写逻辑，直接丢给后厨
    return this.booksService.create(createBookDto);
  }
}
```



## service 服务：后厨主厨
唯一职责：处理核心业务逻辑（搬砖、搞数据、算账）
主厨根本不需要管这个菜是打包带走还是在店里吃，他坐在厨房里，只负责：
 1.查库存：连接数据库（MySQL/Prisma/TypeORM）增删改查。
 2.加工食材：处理业务逻辑（比如密码加密、计算折扣、过滤敏感词）。
 3.调用外部帮工：调用第三方服务（如发送短信、写入 Redis 缓存）。

```javascript
@Injectable() // 声明这是一个可以被注入的“师傅”
export class BooksService {
  // 假设这里注入了数据库实例 prisma
  constructor(private prisma: PrismaService) {}

  async create(createBookDto: CreateBookDto) {
    // 真正的业务逻辑全部写在这里！
    const { title, price } = createBookDto;
    
    // 1. 比如计算个折扣价格
    const discountPrice = price * 0.9; 
    
    // 2. 写入数据库
    return this.prisma.book.create({
      data: { title, price: discountPrice }
    });
  }
}
```

# @Module 模块
参数为对象，对象有四个属性

| **参数名**  | 接收什么类型 | 通俗比喻           | 核心作用与通俗解释                                           |
| ----------- | ------------ | ------------------ | ------------------------------------------------------------ |
| controllers | 数组 `[]`    | 前台服务员         | 注册该模块里的 **控制器**。告<br />诉 Nest 哪些类负责接收 HTTP 请求、<br />匹配路由并返回响应。<br />如果这里不写，你的 API 接口就会报 404。 |
| providers   | 数组 `[]`    | 后厨各岗位的厨师   | 注册该模块里需要用到的 **服务、仓储、策略或工厂**。<br />所有标记了 `@Injectable()` 的类（如 `CatsService`）都要在这里注册，<br />Nest 才会自动帮你 `new` 它们，并实现依赖注入（DI）。 |
| exports     | 数组 `[]`    | 外卖/外借的厨师    | **导出** 本模块内的某些 `provider`。<br />默认情况下，模块内的 Service 是私有的（只能自己内部的 Controller 用）。<br />如果你想让 **其他模块** 也能用你的 `CatsService`，<br />就必须把它放进 `exports` 里。 |
| imports     | 数组 `[]`    | 采购其他分店的物资 | **导入** 其他模块。<br />如果你的模块需要用到别的模块导出的 Service（比如 `CatsModule` 要用 `UserModule` 里的 `UserService` 来查主人信息），<br />就必须把 `UserModule` 放在这里。 |

# Injectable() 注入器 (可注入的)
ai(gemini)解释：
  在它的头顶加上 @Injectable()，相当于给这个类贴上了一个“合格上岗证”。它是去告诉 NestJS 的 IOC 容器（负责帮你自动 new 对象的管家）：“管家，我已经做好准备了，身上也带了标记。以后如果哪个 Controller 或者是别的 Service 需要我，你（NestJS）负责把我 new 出来，然后塞给他们！”
## 🛠️ 那么，到底是谁实现了“自动注入”？
真正触发“把 Service 注入到 Controller”这个动作的，是 Controller 的 constructor（构造函数）。
### 第一步：Service 声明自己“可以被注入”
```javascript
@Injectable() // 1. 贴上上岗证：“我是合法的、可以被别人使用的服务”
export class CatsService {
  findAll() { return ['橙猫', '狸花猫']; }
}
```
### 第二步：Controller 通过 constructor 请求注入
```javascript
@Controller('cats')
export class CatsController {
  // 2. 关键在这里！通过构造函数里的声明：private readonly catsService: CatsService
  // 3. NestJS 看到这里，就会去 providers 里面找到 CatsService 的实例，自动帮你塞进来
  constructor(private readonly catsService: CatsService) {} 
}
```

# 环境变量处理
在 NestJS 中，环境变量通常通过 `@nestjs/config` 模块来处理。这个模块提供了一种简单而强大的方式来管理和访问环境变量，使得应用程序的配置更加灵活和安全。
## 安装 `@nestjs/config` 和 `joi`
```bash
pnpm add @nestjs/config joi
pnpm add -D @types/joi
```
